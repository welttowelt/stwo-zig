At Fib2M the Zig RISC-V Metal lane is 4.083x faster for the prove interval and
3.045x faster for full fresh-process time than its CPU lane. The Rust Cairo
Metal lane is only 0.299x the SIMD prove rate and 0.347x the SIMD full-process

is now a measured generic-backend deficit, not an SN PIE specialization issue.

Native MHz must not be compared across the two VMs. The RISC-V program emits
`5*N-3` RV32IM cycles and uses a lightweight Blake2s benchmark protocol with
blowup 1, FRI fold 1, PoW 10, and three queries. The Cairo program emits
`7*N+16` Cairo opcode cycles and uses the production-like 96-bit protocol with
FRI fold 3, PoW 26, and 70 queries. Absolute cross-VM times are descriptive;
backend speedups are valid only within each VM/protocol pair, and Fib
iterations per second is the workload-normalized cross-VM rate.

The Cairo Metal sweep required fixing a severe generic OODS regression. The
backend bypassed the existing GPU barycentric path and folded columns on the
CPU, while its Metal weight constructor performed scalar QM31 inversions for
every row. A live `/usr/bin/sample` trace placed essentially all CPU time in
`MetalBackend::barycentric_weights`, especially
`coset_vanishing_derivative` and QM31 inversion. The replacement uses STWO's
SIMD batch-inversion weight construction, uploads those exact weights once,
and evaluates a whole column group with a two-stage Metal path: capped
grid-stride partials followed by one reduction threadgroup per column, with one
command submission/readback per group. Fib25k OODS fell from 54.941 seconds to
0.201 seconds and the proof interval fell from 56.764 seconds to about 1.99
seconds without changing the verified proof semantics.

Both Metal labels remain deliberately `hybrid`. In particular, Cairo witness
generation and barycentric weight construction use SIMD host bridges while
commitments, composition, FRI, and grouped OODS evaluation use Metal. The next
generic optimization work should profile the remaining Cairo Metal proof
stages and remove the high fixed/per-log-size costs visible from Fib50k upward;
it should not relabel this as a pure-GPU result.

### Parsed geometry, compact recipes, and base twiddles cut warm setup to 0.401 seconds

The next prepared-geometry slice is live and verified. A heap-stable
`PreparedHostGeometry` owns the parsed schedule, schedule digest, and witness,
feed, relation, fixed-table, and composition bundles. The persistent session
holds four entries in a transactional exact-key LRU keyed by the separate
`PreparedGeometryKey`, so adapted-input geometry cannot accidentally reuse the
narrower resident-state key. Pending geometry is validated before publication
and committed infallibly only after exclusive proof/report publication; any
earlier failure frees the pending entry.

The resident cache now also owns all three compact recipes. Their descriptors
are host-owned and rematerialized after each full-arena reset. Fixed-table,
multiplicity-feed, base-AOT, interaction-AOT, and compact recipe telemetry all
report exact warm hits.

Profiling the old 0.362522792-second broad recipe/prelude interval showed that
only 0.034124958 seconds was attributed recipe construction. The main hidden
work was `populateCommitmentInverseTwiddles`: every request allocated about
256 MiB of host arrays, generated and bit-reversed a 128 MiB tower,
batch-inverted it, copied the 128 MiB inverse bank into `ForwardTwiddles`, and

prepared-state capture; exact hits restore it from the request-start snapshot.
The snapshot grows by exactly 134,217,728 bytes, from 4,371,044,576 to
4,505,262,304 bytes. Later proof phases deliberately overwrite the same arena
binding with forward and inverse banks, and the next request reset restores the
base inverse image.

### Host plans now survive A/B/A resident-geometry switches

The compact canonical arena plan is no longer owned by the capacity-one 26 GB
resident-state entry. `PreparedStateCache` now contains a fixed-slot,
capacity-four transactional `PreparedGeometryCache` for immutable host plans,
while the Metal arena, immutable snapshot, prepared ranges, and base AOT
witness recipe remain capacity one because they own or reference device
storage. A resident switch A -> B clears only B's predecessor device state;
returning B -> A can borrow committed host plan A without rerunning the
multi-second arena coloring pass.

Admission remains fail-closed. A new plan is pending and invisible until the
session has validated the proof/report and commits. A failed pending request
deinitializes the uncommitted plan. A failed hit conservatively evicts only that
active geometry; other committed entries survive. Capacity eviction is a
deterministic four-entry LRU, and plan pointers remain stable for the serial
request lease. Noncanonical and projected requests neither reuse nor evict
committed canonical plans.

exclusive proof/report publication; any later serialization or publication


## Completed resident base subsystem

owned by `CairoWitnessScheduler`. Every component executes one atomic operation:

1. Materialize direct seed data or producer-derived input.
2. Run gather or compact input construction.
3. Run the AOT/native writer.
4. Immediately interpolate its base trace into retained base coefficients.
5. Produce required subcomponent feeds.

2. Base root matches Rust for the exact same SN PIE and parameters.
3. Transcript `[z, alpha]` matches Rust.
4. Interaction claim and interaction root match Rust.
5. Only then execute composition, quotient, FRI, proof assembly, decoding, and
   reference verification.

No later phase should run merely because its kernels compile. A mismatch at an
earlier gate invalidates every later measurement.
