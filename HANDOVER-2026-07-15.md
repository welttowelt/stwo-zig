# Stwo Zig/Metal Cairo Prover Handover

Date: 2026-07-15

## Executive status

The missing large subsystem is now specifically identified: Stwo-Cairo's
per-component resident witness pipeline.

The Zig path still:

- Runs the flat `executeRecordedWitnessGraph` path.
- Materializes lookup slabs during base generation.
- Defers all base interpolation until after the complete witness graph.

Rust instead performs component-local gather/write/interpolate/spill, releases
scratch immediately, and recomputes lookup data during interaction. Until that
ownership model is ported, planned liveness cannot match the writes performed
by the Metal kernels.

The SN PIE is not proven. There is no verified proof and no valid SN PIE MHz
number yet. Do not continue into transcript, quotient, FRI, or proof assembly
until the base-root gates below pass.

## Zig prover optimization ledger

The Cairo work sits on top of substantial prover-wide optimization. Preserve
these implementation choices while replacing the flat Cairo witness path.

### Measured CPU throughput and RSS reduction

The exact RISC-V comparison in `docs/riscv-rust-parity.md` used the same
58,032-byte guest ELF, input, 2,500,157 VM cycles, and M4 Max laptop.

```text
Rust: 9.041 s prove, 0.2765 MHz, 6.94 GB peak RSS
Zig:  1.211 s prove, 2.0650 MHz, 1.53 GB peak RSS
```

For that benchmark, Zig delivered 7.47x Rust throughput and used approximately
78% less peak RSS, or about 4.5x less memory. The main changes were:

- Four-message SIMD Blake2s.
- Hardware-vectorized M31 operations and FFT butterflies.
- Parallel proving phases and parallel PoW grinding.
- Cached PoW prefix hashing.
- Opcode and memory row sharding into log-16 components.
- Removal of protocol-known zero placeholder columns.
- Overlap of independent opcode and infrastructure trace generation.
- Retention of live Merkle data rather than unsafe page discard.

This is a RISC-V implementation diagnostic. Its AIR still has documented
soundness/parity limitations, so do not present it as a Rust-equivalent Cairo
proof result.

### Measured verified Metal throughput

The exact Metal benchmark in `docs/metal-backend-progress.md` uses the same
2,500,157-cycle RISC-V workload and reports a proof accepted by the CPU
verifier. The latest three-run median was:

```text
prove only:  390.3 ms, 6.406 MHz
run + prove: 610.8 ms, 4.093 MHz
```

Against the recorded same-program Rust baseline, the prove-only path was
23.16x faster. Against the preserved Zig CPU median, it was 3.10x faster.

These numbers are evidence for the prover and Metal backend architecture, not
for SN PIE completion. Cairo has different column geometry, interaction work,
and witness ownership, and currently fails before a valid base root.

### CPU and memory optimizations already in the prover

- Contiguous batch buffers replace per-column FFT allocation and improve SIMD
  locality.
- Shared worker pools avoid repeated thread-pool setup and parallelize FFT,
  commitments, quotient work, and AIR accumulation.
- Streaming column commitment bounds temporary memory instead of constructing
  every expanded column simultaneously.
- Lazy quotient evaluation fuses quotient construction with FRI commitment.
- Larger FRI fold steps reduce intermediate allocations.
- mmap-backed storage and explicit page release exist for data whose lifetime
  is genuinely over. Live authentication data is retained or recomputed.
- Implicit zero columns are excluded from materialization and commitment where
  the protocol permits it.

The lower-RSS objective must remain semantic. Never use `MADV_DONTNEED` or
equivalent release on values required for decommitment; that earlier shortcut
caused nondeterministic `RootMismatch` failures.

### Metal residency and bandwidth optimizations already implemented

- Transaction-level `MetalProverEngine` substitution point rather than a
  collection of isolated GPU helper calls.
- Prefix-compatible Blake2s leaves and parent reductions in Metal command
  buffers.
- Mixed-log lifted-column commitment with deterministic ordering.
- Retained Metal Merkle trees and selective opening readback. Full trees are
  not copied back to the CPU.
- Raw-column quotient accumulation on Metal without four CPU-created scaled
  copies of every source column.
- Two-stage sampled evaluation that shares bases by `(log size, point)` and
  avoids a second Zig flattening copy.
- Fused circle IFFT/RFFT tails with 2,048-element threadgroup tiles.
- Contiguous zero-copy transform arenas and GPU blits for column runs.
- Coordinate-major quotient output, removing a 40 MiB compatibility transpose.
- Metal-owned circle-to-line and variable-step FRI folds.
- Metal-owned retained FRI layers and compact decommitment gathers.
- Resident sparse-parent hashing and compact trace/FRI opening assembly.
- Prepared transcript, query mapping, quotient, FRI, and commitment kernels
  with CPU-oracle parity tests.

Relevant measured stage evidence from `docs/metal-backend-progress.md`:

```text
2,106-column synthetic tree: 11.21 billion committed cells/s GPU time
raw FRI quotient kernel:      9.4-13.9 ms
quotient + commit stage:      about 32.2 ms, down from about 478 ms CPU
sampled evaluation:           about 18-24 ms
dominant fused transforms:    about 57.236 ms GPU time
FRI decommit:                 about 1 ms
```

### Cairo-specific memory and residency improvements in this worktree

- Runtime `CairoProofPlan` data replaces compile-time specialization of
  statement-dependent geometry.
- Topological component scheduling replaces name-based lifetime inference.
- Epoch-local physical recoloring prevents future quotient/FRI/decommit
  buffers from inflating the base allocation.
- Scheduled pointer and twiddle bindings replace scratch appended beyond the
  arena.
- Native Metal memory traces avoid CPU materialization and compatibility
  readback for address, big-value, and small-value tables.
- Public-memory seeding and RC9_9 counting update resident multiplicity slabs
  directly with atomic operations.
- Sequential component ticks reflect the executor that exists today, avoiding
  false same-level concurrency and reducing the base peak.
- Retained commitment evaluations are assigned to decommit recomputation rather
  than occupying the base commitment epoch.

Together, the tick-coordinate correction, true base projection, scheduled
scratch, and corrected ownership diagnostics reduced the measured base peak
from approximately 25.72 GiB to 12.22 GiB. This is a 52.5% reduction in the
planned peak, but it is not yet a measured Cairo peak-RSS comparison with Rust.

### Zig implementation quality constraints

Continue following the patterns used by the new modules:

- PIE-dependent plans are ordinary runtime data, not comptime-generated types.
- Use tagged unions and enums for trace parts, stages, and buffer roles.
- Validate exact geometry and canonical ordering at preparation time.
- Own allocator-backed slices explicitly and pair every owner with `deinit`.
- Propagate typed errors and fail closed on missing bindings or parity drift.
- Prepare reusable Metal pipelines and descriptors outside hot execution.
- Keep protocol orchestration visible in `CairoResidentProver`; do not hide
  phase transitions behind backend-global state.
- Specialize Metal kernels where specialization removes branches or output
  materialization, but do not generate the whole proof structure at comptime.

## Exact remaining subsystem

The next implementation should replace the flat witness path with a prepared
per-component pipeline owned by `CairoWitnessScheduler`.

For every component, execute one atomic operation:

1. Materialize direct seed data or producer-derived input.
2. Run gather or compact input construction.
3. Run the AOT/native writer.
4. Immediately interpolate its base trace into retained base coefficients.
5. Produce required subcomponent feeds.
6. Release base trace, temporary input, and producer slabs whose last consumer
   has completed.

Do not retain base `LookupInputs`. The AOT code generation needs separate
output ownership, preferably specialized kernels:

```text
base epoch:        base columns + required subcomponent producer words
interaction epoch: recomputed lookup words only
```

A runtime output mask is possible, but separate specialized Metal entry points
avoid branches and make liveness explicit. The interaction pipeline should
recompute one component's lookup material, immediately execute its relation
instances, and then release that lookup slab.

Native memory and fixed-table components need equivalent scheduler operations:

- Complete multiplicity feeds and public-memory seeding.
- Generate the native memory traces.
- Generate fixed-table base traces after all relevant counts are final.
- Immediately interpolate each native base-trace group.
- Release evaluation-only scratch at the owning tick.

## Why the current path cannot produce a trustworthy root

The arena planner says recorded `BaseTrace` buffers are component-local, but
the CLI calls bulk `interpolateTraceColumns` only after all witness programs.
Those two statements cannot both be true. Earlier component-local base columns
may already have been aliased or overwritten when bulk interpolation reads
them.

Similarly, lookup inputs are written in the base epoch even though they are
only consumed by interaction relations. This creates the approximately 4 GiB
peak and defeats the intended recomputation boundary.

Fix the execution ownership rather than expanding the arena or relaxing
liveness checks.

Enforce these gates in order:

3. Transcript `[z, alpha]` matches Rust.

## Local execution caution

This is a 36 GiB unified-memory M4 Max. Before a resident run, inspect other
large processes:

```sh
ps -axo pid,rss,%mem,command | sort -k2 -nr | head -15
vm_stat | head -15
```

At handover time, a separate Rust `kernel_emit` process was using roughly
6.7 GiB. Do not kill unrelated work, but wait for it to finish before using a
SIGKILL or empty output as evidence about the Metal backend. Machine contention
and the flat-pipeline design defect are separate facts.

## Definition of done

This handover is complete only when the team can produce one reference-verified
SN PIE proof and report honest VM cycles per second from that verified path.
Synthetic Fibonacci MHz and unverified commitment timings do not satisfy the
goal.
